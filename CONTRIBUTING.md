# Contributing to NOOP

Welcome, and thank you for your interest in contributing. NOOP is an offline-first companion app
for WHOOP 4.0 and 5.0 straps — no servers, no accounts, no data leaving the device.

The **full contributing guide** lives in [`docs/CONTRIBUTING.md`](docs/CONTRIBUTING.md). This file
is a quick-start reference so you can get oriented fast.

---

## Quick start

### macOS / Swift

```bash
# 1. Fork and clone
git clone https://github.com/<you>/noop.git && cd noop

# 2. Install XcodeGen and generate the project
brew install xcodegen
xcodegen generate

# 3. Build and test (no signing required)
xcodebuild -project Strand.xcodeproj -scheme Strand \
  -sdk macosx -configuration Debug \
  CODE_SIGNING_ALLOWED=NO CODE_SIGNING_REQUIRED=NO \
  build test

# 4. For per-package iteration (fastest loop)
cd Packages/WhoopProtocol && swift build && swift test
```

### Android / Kotlin

```bash
# 1. Fork and clone
git clone https://github.com/<you>/noop.git && cd noop/android

# 2. Build debug APK (JDK 17 required)
./gradlew assembleFullDebug

# 3. Run unit tests
./gradlew testFullDebugUnitTest
```

## What CI checks

Every PR runs the following GitHub Actions workflows automatically.

| Workflow | Trigger | What it does |
|---|---|---|
| **Android CI** | Changes under `android/` | Builds `assembleFullDebug`, runs `testFullDebugUnitTest`, uploads APK artifact |
| **macOS CI** | Changes under `Strand/`, `StrandTests/`, `Packages/`, `project.yml` | Installs XcodeGen 2.45, generates project, runs `xcodebuild build test` with `CODE_SIGNING_ALLOWED=NO` |

**All CI checks must pass before a PR is merged.** If CI fails on your PR, fix the issue rather
than bypassing it. Do not commit `Strand.xcodeproj/` — it is gitignored and regenerated by CI.

## Submitting a PR

1. Open one PR per concern (protocol, schema, UI, and Android changes in separate PRs where practical).
2. Fill in the [PR template](.github/PULL_REQUEST_TEMPLATE.md): description, change type, checklist.
3. For anything on the BLE path, state what you tested on real hardware in the PR description.
4. For analytics changes, cite the method and include a test.
5. For UI changes, confirm all colors, fonts, and spacing come from `StrandDesign` tokens only.

See [`docs/CONTRIBUTING.md`](docs/CONTRIBUTING.md) for full conventions including the BLE safety
contract, design system rules, how to add metrics/screens/commands, and the commit style guide.
